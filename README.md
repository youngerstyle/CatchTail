# CatchTail

CatchTail is a Codex plugin project. The repository root is installable as a
single plugin, and the same plugin is also mirrored under `plugins/catchtail/`
for repo-local marketplace testing.

CatchTail is a lightweight local interaction sidecar for long-running Codex
sessions. It does not patch Codex internals. It uses Codex hooks plus a small
local console so you can continue sending messages, task input, files, and a
final milestone signal while the agent loop is still alive.

The core model is deliberately small:

- `queue.json` is the current unclaimed input queue.
- `session.jsonl` is the append-only session history.
- `state.json` stores interactive mode and milestone state.
- Codex hook `session_id` isolates each Codex session.

The implementation is Node.js-only and uses no third-party packages.

## Plugin Layout

The root directory is the canonical plugin:

```text
.codex-plugin/plugin.json        Codex plugin manifest
hooks.json                       Plugin hook declaration
skills/catchtail-interactive/    Skill bundled with the plugin
bin/                             CLI entrypoint
src/                             Runtime, hook, CLI, and console implementation
```

For local marketplace development, the same plugin is mirrored at:

```text
plugins/catchtail/
.agents/plugins/marketplace.json
```

This makes the repository understandable in both common shapes:

- Single-plugin repo: install the repository root directly.
- Marketplace-style repo: load `plugins/catchtail` through the marketplace file.

## What It Builds

- A Codex hook adapter for `UserPromptSubmit` and `Stop`.
- A local state directory at `.catchtail/sessions/<session_id>/`.
- A project-local skill at `.agents/skills/catchtail-interactive/`.
- A browser console for sending messages, uploading files, previewing images,
  opening uploaded files with the system default app, and stopping the queue.
- A starter `AGENTS.catchtail.md` protocol that can be merged into the active
  `AGENTS.md`.

## Quick Start

### Install From A Plugin Checkout

Clone or install this repository as a Codex plugin, then run the project
installer in the target project:

```powershell
node .\scripts\install.mjs <target-project>
```

For local development in this repository, run:

```powershell
node .\bin\catchtail.js init
node .\bin\catchtail.js serve
```

For the mirrored marketplace plugin shape, run from the target project root:

```powershell
node .\plugins\catchtail\scripts\install.mjs
node .\plugins\catchtail\bin\catchtail.js serve
```

Open the printed local URL, then start Codex in this project and say:

```text
启动交互式工作流
```

After `init`, review `.codex/hooks.json` with Codex `/hooks` and trust the hook
if prompted. Codex discovers project-local hooks from `<repo>/.codex/hooks.json`
when the project config layer is trusted.

## Files

```text
src/core.js              State machine and hook behavior
src/hook.js              stdin/stdout Codex hook adapter
src/server.js            Local web console and JSON API
src/cli.js               init / serve / status / message commands
hooks/catchtail-hook.js  Hook entrypoint used by Codex
AGENTS.catchtail.md      Protocol text generated by init
.agents/skills/...       Skill generated by init

.catchtail/sessions/<session_id>/state.json     Runtime session state
.catchtail/sessions/<session_id>/queue.json     Current unclaimed queue
.catchtail/sessions/<session_id>/session.jsonl  Append-only history
```

Manual CLI calls use the `default` session unless you pass
`--session <codex-session-id>`.

## Hook Behavior

- `UserPromptSubmit`: enables interactive mode when the prompt contains
  `启动交互式工作流`, then injects developer context for the protocol.
- `Stop`: if interactive mode is active and the milestone is not completed,
  waits briefly on the local sidecar when the queue is empty, then returns
  `decision: "block"` with a continuation prompt.

No approval, pause, tool-audit, or tool-blocking hook is installed by the core
workflow. Codex's normal permission and sandbox rules still apply.

## Console API

- `GET /api/state`
- `GET /api/queue`
- `POST /api/queue` with `{ "kind": "message|task|question", "body": "...", "files": [] }`
- `POST /api/queue/claim`
- `POST /api/queue/cancel` with `{ "id": "<message-id>", "reason": "..." }`
- `POST /api/queue/complete` with `{ "id": "<message-id>", "response": "..." }`
- `GET /api/session`
- `GET /api/file?path=<uploaded-file-path>`
- `GET /api/wait?timeoutMs=540000`
- `POST /api/messages` with `{ "kind": "message|task|question", "body": "...", "files": [] }`
- `POST /api/files` with multipart `files`
- `POST /api/open` with `{ "path": "<uploaded-file-path>" }`
- `POST /api/milestone` with `{ "milestone": "incomplete|completed" }`

`/api/wait` is used by the `Stop` hook for local long-poll waiting. It resolves
when a new message or milestone update arrives.

`/api/file` and `/api/open` are restricted to files saved under
`.catchtail/uploads/`. The console stores text drafts in browser localStorage
per Codex session and saves attachments to the sidecar before preserving them in
the draft.

The `/api/queue*` endpoints are the stable third-party integration surface.
They include CORS headers so local tools, browser extensions, or companion UIs
can push, inspect, claim, and complete queue items without driving the console
DOM.
Pass `?sessionId=<id>` to target a specific Codex session. The queue API treats
`sessionId` as required and returns `400` when it is omitted.

## Agent Message Processing

When CatchTail reports pending input, the agent should explicitly claim and
complete it:

```powershell
node .\bin\catchtail.js claim
node .\bin\catchtail.js complete <message-id> "short response"
node .\bin\catchtail.js wait
```

`claim` removes the item from `queue.json`. `complete` appends the handling
result to `session.jsonl`. In interactive mode, the agent should call `wait`
again after `complete` while the milestone is still incomplete. This keeps the
current turn alive instead of relying on a later model turn after `Stop`.

For manual work on a specific session:

```powershell
node .\bin\catchtail.js --session <codex-session-id> status
node .\bin\catchtail.js --session <codex-session-id> claim
```

## Idle Waiting

CatchTail avoids model-side polling while idle. When the `Stop` hook fires and
there is no queued input, the hook calls:

```text
GET http://127.0.0.1:3787/api/wait?timeoutMs=540000
```

If the sidecar wakes because a message arrives, `Stop` returns a continuation
prompt that includes the pending input. If the sidecar is unavailable or the
wait times out, the hook falls back to a continuation prompt so the session
remains recoverable.

The preferred live workflow is still `claim -> complete -> wait` inside the
current agent turn. The `Stop` hook is a fallback boundary, not the main message
processing loop.

## Testing

PowerShell may block `npm.ps1` on some machines. The direct command is:

```powershell
node --test
```

## Current Limits

- This is an external sidecar, not a Codex core-loop patch.
- New user messages enter Codex at hook boundaries, especially `Stop`.
- Uploaded files are copied under `.catchtail/uploads/`; Codex still needs
  filesystem access to inspect them directly.
- If the quota-exhausted environment cannot run hooks at all, CatchTail cannot
  force Codex to re-enter the loop.
